import random

# Simulating network nodes with their clock times (in seconds)
class Node:
    def __init__(self, node_id, clock_time):
        self.node_id = node_id
        self.clock_time = clock_time

    def get_time(self):
        return self.clock_time

    def adjust_time(self, offset):
        self.clock_time += offset

    def __repr__(self):
        return f"Node {self.node_id}: {self.clock_time:.2f} sec"


class Coordinator(Node):
    def synchronize_clocks(self, nodes):
        print("Polling times from all nodes...")
        # times[node_id] stores node offset relative to coordinator clock.
        times = {}

        # Step 1: Collect each node's time offset relative to coordinator.
        for node in nodes:
            if node.node_id == self.node_id:
                continue
            # Simulate network delay and assume symmetric one-way delay.
            delay = random.uniform(0.1, 0.5)
            node_time = node.get_time() + delay / 2  # assume symmetric delay
            # Positive offset means node appears ahead of coordinator.
            times[node.node_id] = node_time - self.clock_time
            print(f"Node {node.node_id} time: {node.get_time():.2f}, adjusted for delay: {node_time:.2f}")

        # Coordinator's own offset to itself is always zero.
        times[self.node_id] = 0.0

        # Step 2: Compute group average offset used as consensus time shift.
        average_offset = sum(times.values()) / len(times)
        print(f"\nCalculated average offset: {average_offset:.2f} seconds")

        # Step 3: Send per-node correction so all clocks converge.
        for node in nodes:
            # Node-specific correction = target offset - node's current offset.
            offset = average_offset - times[node.node_id]
            node.adjust_time(offset)
            print(f"Adjusting Node {node.node_id} by {offset:+.2f} seconds")

        print("\nSynchronization complete.\n")


# --- Example usage ---
if __name__ == "__main__":
    # Create nodes with different clock times
    nodes = [
        Coordinator(0, 100.0),   # Coordinator
        Node(1, 102.0),
        Node(2, 98.5),
        Node(3, 101.2),
        Node(4, 99.8),
    ]

    print("Initial times:")
    for node in nodes:
        print(node)

    coordinator = nodes[0]
    coordinator.synchronize_clocks(nodes)

    print("Final times after synchronization:")
    for node in nodes:
        print(node)

## рҹҺ“ ORAL EXAM QUESTIONS WITH ANSWERS

### Basic Level Questions
1. **What is clock synchronization and why is it important in distributed systems?**
   
   **Answer**:
   Clock synchronization means keeping the clocks of all machines in a distributed system close to one another.
   It matters because:
   - event ordering becomes clearer
   - logs from different nodes can be compared
   - replicated databases can decide which update is newer
   - timeout-based protocols behave more predictably
   
   If clocks differ a lot, one node may think an event happened "in the future" or "in the past," which breaks debugging and coordination.

2. **What is the Berkeley Algorithm?**
   
   **Answer**:
   The Berkeley Algorithm is a distributed clock synchronization method where one node acts as a coordinator.
   The coordinator collects time values from all machines, computes an average, and tells each node how much to adjust its clock.
   
   It is useful when there is no external time source like an atomic clock and the goal is to make all local clocks agree with each other.

3. **What is the role of the coordinator in Berkeley Algorithm?**
   
   **Answer**:
   The coordinator is responsible for:
   - polling all nodes for their current time
   - estimating network delay when collecting times
   - computing the average or corrected average time
   - sending adjustment values back to each node
   - ignoring faulty or extreme outlier clocks if needed
   
   In practice, any node can be chosen as coordinator, but usually the one with the best connectivity or least load is preferred.

4. **Explain the three main steps of Berkeley Algorithm.**
   
   **Answer**:
   The algorithm works in three phases:
   - **Collection phase**: coordinator asks each node for its local time
   - **Calculation phase**: coordinator computes average time and the offset for each node
   - **Distribution phase**: coordinator sends adjustment values to every node
   
   After that, each node updates its local clock by the amount requested.

5. **What is clock skew and clock drift?**
   
   **Answer**:
   Clock skew is the difference between two clocks at a given moment.
   Clock drift is the rate at which one clock slowly moves away from another over time.
   
   Example:
   - If Node A shows 10:00 and Node B shows 10:02, the skew is 2 minutes.
   - If Node B gains 1 second every 10 minutes, that is drift.
   
   Drift happens because physical clocks are imperfect and their oscillators are not identical.

### Intermediate Level Questions
6. **How does Berkeley Algorithm handle network delays?**
   
   **Answer**:
   Berkeley estimates delay by assuming the round-trip delay is roughly symmetric.
   If a request takes D milliseconds to go to a node and come back, it assumes about D/2 was spent one way.
   
   This estimate is used to correct the reported clock value before computing the average.

7. **Why is average time used instead of median in Berkeley?**
   
   **Answer**:
   The average is used because the goal is to minimize the total adjustment across the whole group.
   It gives a single consensus time around which all nodes can converge.
   
   The median can resist outliers better, but Berkeley traditionally uses averaging and then may discard obvious faulty clocks before averaging.

8. **How does Berkeley Algorithm differ from NTP (Network Time Protocol)?**
   
   **Answer**:
   Berkeley is a closed-group synchronization method where one node coordinates the others and the group agrees on a common internal time.
   NTP synchronizes clocks using external reference servers and a hierarchy of time sources.
   
   Main difference:
   - Berkeley: good for internal agreement inside a cluster
   - NTP: good for keeping time close to real-world UTC

9. **What are the time complexities of Berkeley Algorithm?**
   
   **Answer**:
   Message complexity is O(n) because the coordinator must contact each of the n nodes.
   Time complexity is also effectively O(n) for a single synchronization round, although the theoretical question paper sometimes writes it as O(log n) when discussing multi-level hierarchical synchronization.
   
   In a basic flat network, the important fact is that every node is contacted once.

10. **How would you handle a faulty coordinator in Berkeley?**
    
    **Answer**:
    If the coordinator fails, a new coordinator must be elected.
    Common approaches:
    - choose the next healthy node in a predefined order
    - run a leader election algorithm like Bully or Ring
    - keep a backup coordinator that can take over
    
    The key requirement is that only one coordinator should be active at a time.

### Advanced Level Questions
11. **Design a Byzantine-fault-tolerant clock synchronization algorithm.**
    
    **Answer**:
    A Byzantine-fault-tolerant version must handle malicious nodes that send wrong time values on purpose.
    A practical design is:
    - collect times from all nodes
    - verify messages using authentication
    - discard values that are too far from the majority
    - compute the average only from trusted nodes
    - use quorum-based agreement when choosing the final time

12. **Explain how Berkeley Algorithm maintains global time consistency.**
    
    **Answer**:
    Berkeley maintains consistency by repeatedly forcing all clocks toward the group average.
    Even if small drift exists, repeated synchronization rounds keep the maximum difference between clocks bounded.
    
    The system does not guarantee perfect real-world time, but it does guarantee that all participating nodes stay close to each other.

13. **Compare Berkeley with Cristian's Algorithm.**
    
    **Answer**:
    Cristian's Algorithm asks a time server for the current time and adjusts based on the reply.
    Berkeley instead asks all nodes, computes a group average, and synchronizes the whole cluster internally.
    
    Comparison:
    - Cristian: server-centered, useful when accurate external time exists
    - Berkeley: group-centered, useful when internal agreement is enough

14. **How would you implement Berkeley Algorithm in a large-scale system?**
    
    **Answer**:
    For large systems, use hierarchical synchronization.
    
    Example:
    - local clusters synchronize with local coordinators
    - local coordinators synchronize with a higher-level coordinator
    - the top-level coordinator propagates adjustments back down
    
    This reduces communication overhead and keeps synchronization scalable.

15. **Design a distributed clock synchronization system with dynamic nodes.**
    
    **Answer**:
    The system should support node join and leave events.
    
    Design:
    - when a node joins, it reports its clock during the next synchronization round
    - when a node leaves, the coordinator simply excludes it from the next average
    - if a node is unstable, mark it as suspicious and ignore it temporarily
    
    This keeps the system stable even when membership changes.

### Scenario-Based Questions
16. **How would you detect and remove faulty nodes in Berkeley synchronization?**
    
    **Answer**:
    Use outlier detection.
    If a node's reported time is too far from the rest of the cluster, mark it as faulty.
    
    Example rule:
    - compute average time
    - if |node_time - average| > threshold, exclude node
    - recompute average without those nodes
    
    This prevents one bad clock from dragging the whole group away from correct time.

17. **Design a real-time system that uses Berkeley for synchronization.**
    
    **Answer**:
    A real-time monitoring system can use Berkeley to keep log timestamps and sensor events aligned across all machines.
    
    Requirements:
    - low drift between nodes
    - periodic synchronization rounds
    - quick recovery when a node joins or fails
    - bounded error tolerance
    
    Berkeley is useful here because the system mainly needs internal consistency, not exact UTC.

18. **Compare Berkeley vs Lamport logical clocks for ordering events.**
    
    **Answer**:
    Berkeley uses physical time and tries to make real clocks match.
    Lamport logical clocks do not use real time; they only preserve causality ordering.
    
    Use Berkeley when:
    - you need timestamps that look like real time
    - logs should be comparable across nodes
    
    Use Lamport clocks when:
    - you only care about event ordering
    - physical clock accuracy is not important

19. **Implement Berkeley with variable network delays.**
    
    **Answer**:
    The coordinator should measure delay for each node individually.
    
    Strategy:
    - timestamp request send time
    - timestamp reply receive time
    - estimate one-way delay as half of the RTT
    - correct the node's reported time using that estimate
    
    If delays are asymmetric, use historical averages or discard unstable links.

20. **Design a hybrid synchronization system combining Berkeley and NTP.**
    
    **Answer**:
    A hybrid system can use NTP for approximate external UTC alignment and Berkeley for internal group consistency.
    
    Design:
    - NTP synchronizes the cluster to a rough global time source
    - Berkeley keeps all nodes close to each other inside the cluster
    - if NTP becomes unavailable, Berkeley still preserves internal agreement
    
    This gives both real-world correctness and good local consistency.

### 2. **Berkeley Algorithm**

#### **Definition**
Berkeley Algorithm is a clock synchronization algorithm where a coordinator node collects clock values from all other nodes, calculates the average, and distributes adjustment values to synchronize all clocks.

#### **Historical Context**
- **Invented**: 1985 at University of California, Berkeley
- **Named after**: Where the algorithm was developed
- **Purpose**: Synchronize internal clocks in distributed systems
- **Scope**: Works within closed networks (not for external synchronization)

#### **Algorithm Overview**

**Key Characteristics**:
- вң… **Centralized**: One coordinator node
- вң… **Synchronous**: Nodes respond to coordinator queries
- вң… **Averaging**: Uses mean of collected times
- вң… **Simple**: Easy to understand and implement
- вң… **Fault-tolerant**: Can handle some faulty clocks
- вң… **Probabilistic**: Works based on statistical properties

**Limitations**:
- вқҢ Coordinator dependency: Coordinator failure breaks system
- вқҢ No external time: Cannot synchronize with UTC
- вқҢ Assumes symmetric delays: Network must be relatively stable
- вқҢ Requires quorum: Needs most nodes working

### 3. **Understanding Network Delays**

#### **Symmetric Delay Assumption**

**In Berkeley Algorithm**:
```
Network Packet Journey:
Coordinator в”Җв”Җ[Delay D]в”Җв”Җв–ә Node
           в—„в”Җв”Җ[Delay D]в”Җв”Җв”Җв”Җ
(Assumes same delay in both directions)

Actual Time Calculation:
- Coordinator sends request at time T0
- Node receives at T0 + D
- Node replies immediately with its clock time C
- Coordinator receives at T0 + 2D
- Delay estimate: (T0 + 2D - T0) / 2 = D
- Adjusted node time: C + D/2 (accounts for incoming delay)
```

**Why Adjust by D/2?**
```
Node's clock shows: C
But it's actually at time: C + D/2 (relative to coordinator)
                                вҶ‘
                    Account for message travel time
```

#### **Example with Real Numbers**

```
Scenario: Coordinator clock = 100.0 seconds

Node 1:
  - Clock shows: 102.0
  - Network delay: 0.2 seconds
  - Adjusted time: 102.0 + 0.1 = 102.1
  - Offset from coordinator: 102.1 - 100.0 = +2.1

Node 2:
  - Clock shows: 98.5
  - Network delay: 0.3 seconds
  - Adjusted time: 98.5 + 0.15 = 98.65
  - Offset from coordinator: 98.65 - 100.0 = -1.35

Node 3:
  - Clock shows: 101.2
  - Network delay: 0.25 seconds
  - Adjusted time: 101.2 + 0.125 = 101.325
  - Offset from coordinator: 101.325 - 100.0 = +1.325

Coordinator:
  - Clock shows: 100.0
  - Offset: 0.0

Average offset: (2.1 - 1.35 + 1.325 + 0.0) / 4 = 0.519

Adjustments:
  Node 1: 0.519 - 2.1 = -1.581 seconds (move back)
  Node 2: 0.519 - (-1.35) = +1.869 seconds (move forward)
  Node 3: 0.519 - 1.325 = -0.806 seconds (move back)
  Coordinator: 0.519 - 0.0 = +0.519 seconds (move forward)

Final times (after adjustment):
  Node 1: 102.0 - 1.581 = 100.419
  Node 2: 98.5 + 1.869 = 100.369
  Node 3: 101.2 - 0.806 = 100.394
  Coordinator: 100.0 + 0.519 = 100.519
  
All synchronized to ~100.4 seconds! вң“
```

### 4. **Berkeley vs Other Synchronization Algorithms**

#### **A. Berkeley Algorithm vs NTP (Network Time Protocol)**

| Aspect | Berkeley | NTP |
|--------|----------|-----|
| **Approach** | Centralized (coordinator) | Hierarchical (stratum levels) |
| **Time Source** | Internal to network | External (atomic clock) |
| **Accuracy** | 1-10 milliseconds | Microseconds (stratum 1) |
| **Scalability** | 100s of nodes | Millions of nodes |
| **Fault Tolerance** | Moderate (coordinator failure) | High (hierarchical) |
| **Purpose** | Internal sync | External sync (UTC) |
| **Implementation** | Simple | Complex |

#### **B. Berkeley vs Cristian's Algorithm**

| Aspect | Berkeley | Cristian's |
|--------|----------|-----------|
| **Model** | Broadcast model | Client-server model |
| **Communication** | One-to-many | One-to-one |
| **Delay Handling** | Symmetric assumption | Measures round-trip |
| **Synchronization** | Average of all clocks | Time server + client |
| **Fault Tolerance** | Voting on multiple servers | Depends on server |

#### **C. Berkeley vs Lamport Clocks**

| Aspect | Berkeley | Lamport |
|--------|----------|---------|
| **Type** | Physical clock sync | Logical clock |
| **Purpose** | Actual time sync | Event ordering |
| **Accuracy** | Milliseconds | No absolute time |
| **Memory** | Minimal | One counter per node |
| **When Needed** | Absolute time critical | Causality critical |

```

в””в”Җ Round 3: Coordinator sends adjustments to n-1 nodes

Total: 3(n-1) = O(n) messages

For 1000 nodes: 2997 messages per sync cycle
```

#### **Time Complexity**
```
Berkeley Algorithm Time:
в”ңв”Җ Parallel communication: O(log n) rounds
в”Ӯ  (with pipelined messages)
в”ңв”Җ Coordinator calculation: O(n) computation
в””в”Җ Total: O(n) dominated by communication

With hierarchical Berkeley:
в”ңв”Җ Multiple coordinators per region
в”ңв”Җ Synchronize regions: O(log n)
в”ңв”Җ Synchronize nodes within region: O(log n)
в””в”Җ Total: O(log n) hierarchical depth
```

### **Complete Program: berkley.py**

```python
import random

# Simulating network nodes with their clock times (in seconds)
class Node:
    def __init__(self, node_id, clock_time):
        self.node_id = node_id
        self.clock_time = clock_time

    def get_time(self):
        return self.clock_time

    def adjust_time(self, offset):
        self.clock_time += offset

    def __repr__(self):
        return f"Node {self.node_id}: {self.clock_time:.2f} sec"


class Coordinator(Node):
    def synchronize_clocks(self, nodes):
        print("Polling times from all nodes...")
        times = {}

        # Step 1: Collect time differences
        for node in nodes:
            if node.node_id == self.node_id:
                continue
            # Simulate network delay
            delay = random.uniform(0.1, 0.5)
            node_time = node.get_time() + delay / 2  # assume symmetric delay
            times[node.node_id] = node_time - self.clock_time
            print(f"Node {node.node_id} time: {node.get_time():.2f}, adjusted for delay: {node_time:.2f}")

        # Add coordinator itself
        times[self.node_id] = 0.0

        # Step 2: Compute average offset (excluding outliers if needed)
        average_offset = sum(times.values()) / len(times)
        print(f"\nCalculated average offset: {average_offset:.2f} seconds")

        # Step 3: Send adjustment to each node
        for node in nodes:
            offset = average_offset - times[node.node_id]
            node.adjust_time(offset)
            print(f"Adjusting Node {node.node_id} by {offset:+.2f} seconds")

        print("\nSynchronization complete.\n")


# --- Example usage ---
if __name__ == "__main__":
    # Create nodes with different clock times
    nodes = [
        Coordinator(0, 100.0),   # Coordinator
        Node(1, 102.0),
        Node(2, 98.5),
        Node(3, 101.2),
        Node(4, 99.8),
    ]

    print("Initial times:")
    for node in nodes:
        print(node)

    coordinator = nodes[0]
    coordinator.synchronize_clocks(nodes)

    print("Final times after synchronization:")
    for node in nodes:
        print(node)
```

```
в”ңв”Җ Require majority agreement
в”ңв”Җ Exclude minority outliers
в””в”Җ Fault tolerance: tolerates f faulty clocks if n > 3f
```

## рҹ“Ҡ BERKELEY ALGORITHM PROPERTIES

### **Correctness**
```
вң“ All nodes converge to same time (within acceptable bound)
вң“ Average is optimal for symmetric delay model
вң“ Works with any number of nodes
```

### **Fault Tolerance**
```
вң“ Can tolerate f faulty clocks if:
  - Using median: f < n/2
  - Using mean with outlier removal: f < 3n/10
  
вң— Cannot tolerate:
  - Coordinator failure (need failover)
  - Byzantine (malicious) nodes without modifications
```

### **Complexity**
```
Message Complexity: O(n)
  в”ңв”Җ Requests: n-1
  в”ңв”Җ Responses: n-1
  в””в”Җ Adjustments: n-1
  в””в”Җ Total: 3(n-1) messages

Time Complexity: O(log n)
  в””в”Җ Can be pipelined/parallelized

Space Complexity: O(n)
  в””в”Җ Store all node times
```

### **Convergence**
```
Convergence Speed: Linear in synchronization frequency
  в”ңв”Җ More frequent sync: tighter accuracy
  в”ңв”Җ Less frequent sync: wider clock skew
  в””в”Җ Typical: every 10-60 seconds

Steady State:
  в””в”Җ All clocks within Вұ10ms of each other
     (in good network conditions)
```

## рҹ’Ў PRACTICAL CONSIDERATIONS

### **When to Use Berkeley Algorithm**
вң… Internal cluster synchronization (not UTC)
вң… Small to medium networks (< 1000 nodes)
вң… Relatively stable networks
вң… Systems can tolerate brief desynchronization
вң… Don't need external time reference

### **When NOT to Use Berkeley Algorithm**
вқҢ Absolute time required (use NTP)
вқҢ Very large networks (use hierarchical)
вқҢ Highly unreliable networks
вқҢ Byzantine-fault scenarios
вқҢ Real-time systems (use atomic clocks)

### **Optimization Strategies**
```
1. Hierarchical Berkeley
   в”ңв”Җ Divide nodes into groups
   в”ңв”Җ Each group synchronized independently
   в””в”Җ Groups synchronized to master

2. Adaptive Synchronization
   в”ңв”Җ Sync frequency based on measured drift
   в”ңв”Җ More sync when clocks drift apart
   в””в”Җ Less sync when clocks stay together

3. Predicted Correction
   в”ңв”Җ Predict future drift
   в”ңв”Җ Apply preventive adjustment
   в””в”Җ Reduce frequency of full synchronization

4. Hybrid Approach
   в”ңв”Җ Berkeley for internal sync
   в”ңв”Җ Occasional NTP for external sync
   в””в”Җ Combined accuracy/simplicity benefit
```

## рҹҺҜ KEY TAKEAWAYS

1. **Berkeley Algorithm synchronizes clocks through averaging**
   - Coordinator collects all times
   - Calculates average offset
   - Sends individual adjustments

2. **Network delay must be estimated**
   - Symmetric delay assumption
   - Half of round-trip time
   - Accounts for message travel

3. **Fault tolerance through statistical methods**
   - Outlier detection
   - Voting/consensus
   - Graceful degradation

4. **Scalability is O(n) messages**
   - Linear in number of nodes
   - Practical for moderate sizes
   - Hierarchical for large systems

5. **Convergence is guaranteed**
   - All clocks move toward average
   - Synchronization improves each cycle
   - Steady state achieved
